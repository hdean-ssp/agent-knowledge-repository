# Requirements Document

## Introduction

The Agent Knowledge Repository is a Python CLI tool and accompanying framework that enables AI agents to commit, retrieve, and update knowledge in a shared or per-user repository. It serves as a living wiki of agent learnings across all interactions, backed by sqlite-vec (a SQLite vector search extension) for semantic search over structured documentation artifacts. Embeddings are generated locally using a lightweight ONNX-based embedding library, requiring no external APIs or GPU hardware. The system is designed for use in large, legacy codebases but is general-purpose enough for any AI agent workflow. All dependencies are pip-installable with no system-level packages required beyond the Python runtime available natively on RHEL 9. Steering files enforce consistent usage patterns and integrate with existing agent configurations to trigger knowledge operations automatically.

## Glossary

- **CLI_Tool**: The Python command-line interface application that AI agents invoke to commit, fetch, update, and manage knowledge artifacts in the repository.
- **Knowledge_Artifact**: A structured, formatted documentation unit stored in the repository, representing a discrete piece of knowledge an agent has learned or recorded.
- **Vector_Store**: The sqlite-vec-backed vector storage backend that stores Embeddings of Knowledge_Artifacts in a SQLite database and supports semantic similarity search for retrieval.
- **Repository**: The persistent storage location (shared or per-user) containing the Vector_Store and all Knowledge_Artifacts.
- **Shared_Repository**: A Repository instance accessible by all agents and users on a single system, stored at a configurable system-level path.
- **User_Repository**: A Repository instance scoped to a single agent or user, stored within the user's `.kiro/` directory.
- **Steering_File**: A markdown or configuration file that provides rules, workflows, and triggers for AI agents, guiding them to use the CLI_Tool at appropriate moments.
- **Embedding**: A numerical vector representation of a Knowledge_Artifact's content, generated by the ONNX_Embedding_Model and stored in the Vector_Store for semantic similarity search.
- **Agent**: An AI assistant or automated process that interacts with the Repository through the CLI_Tool.
- **Knowledge_Schema**: The defined structure and required fields for a Knowledge_Artifact, ensuring consistency across all entries.
- **Target_Platform**: The operating system and environment the CLI_Tool is designed to run on, specifically Red Hat Enterprise Linux 9 (RHEL 9) with Python 3.9+ (default) or Python 3.11/3.12 (available as app streams).
- **Embedded_Vector_Store**: A Vector_Store implementation using sqlite-vec that runs in-process within the Python application and persists data to local SQLite database files, requiring no separate database server or daemon.
- **sqlite-vec**: A zero-dependency C extension for SQLite that adds vector search capabilities (distance functions, vector indexing). It is pip-installable via the `sqlite-vec` package and requires no system-level package installation on the Target_Platform.
- **ONNX_Embedding_Model**: A lightweight, ONNX-runtime-based embedding library (e.g., fastembed or light-embed) used to generate Embeddings locally. It runs on CPU without requiring PyTorch, TensorFlow, GPU hardware, or external API keys, and is pip-installable.
- **Standard_Library**: The set of modules and packages included with the Python 3.9+ runtime, requiring no additional installation beyond the Python interpreter itself.
- **Language_Package_Manager**: pip, the Python package installer, used to install all CLI_Tool dependencies without requiring system-level package installation (e.g., no `dnf install` or `yum install`).

## Requirements

### Requirement 1: Commit Knowledge

**User Story:** As an AI agent, I want to commit new knowledge to the repository, so that learnings from my interactions are preserved for future retrieval.

#### Acceptance Criteria

1. WHEN the Agent invokes the CLI_Tool with an `akr-commit` command and a valid Knowledge_Artifact payload, THE CLI_Tool SHALL store the Knowledge_Artifact in the Repository and generate an Embedding in the Vector_Store.
2. WHEN the Agent invokes the CLI_Tool with an `akr-commit` command and a payload that does not conform to the Knowledge_Schema, THE CLI_Tool SHALL reject the commit and return a descriptive validation error message.
3. WHEN a Knowledge_Artifact is successfully committed, THE CLI_Tool SHALL return a unique identifier for the committed artifact.
4. IF the Repository is unreachable or write fails, THEN THE CLI_Tool SHALL return an error message indicating the failure reason without crashing.

### Requirement 2: Fetch Knowledge

**User Story:** As an AI agent, I want to fetch relevant knowledge from the repository using semantic search, so that I can leverage prior learnings in my current interaction.

#### Acceptance Criteria

1. WHEN the Agent invokes the CLI_Tool with an `akr-fetch` command and a natural-language query, THE CLI_Tool SHALL return the top-N most semantically similar Knowledge_Artifacts from the Vector_Store, where N is configurable and defaults to 5.
2. WHEN the Agent invokes the CLI_Tool with an `akr-fetch` command and a query that matches no Knowledge_Artifacts above the similarity threshold, THE CLI_Tool SHALL return an empty result set with a message indicating no relevant knowledge was found.
3. WHEN the Agent invokes the CLI_Tool with an `akr-fetch` command, THE CLI_Tool SHALL return results within 2 seconds for repositories containing up to 10,000 Knowledge_Artifacts.
4. THE CLI_Tool SHALL include a relevance score with each returned Knowledge_Artifact in fetch results.

### Requirement 3: Update Knowledge

**User Story:** As an AI agent, I want to update existing knowledge artifacts, so that the repository stays current as understanding evolves.

#### Acceptance Criteria

1. WHEN the Agent invokes the CLI_Tool with an `akr-update` command, a valid artifact identifier, and an updated Knowledge_Artifact payload, THE CLI_Tool SHALL replace the existing artifact content and regenerate the Embedding in the Vector_Store.
2. WHEN the Agent invokes the CLI_Tool with an `akr-update` command and an artifact identifier that does not exist in the Repository, THE CLI_Tool SHALL return an error indicating the artifact was not found.
3. WHEN the Agent invokes the CLI_Tool with an `akr-update` command and a payload that does not conform to the Knowledge_Schema, THE CLI_Tool SHALL reject the update and return a descriptive validation error message.
4. WHEN a Knowledge_Artifact is successfully updated, THE CLI_Tool SHALL record the previous version in an audit trail with a timestamp.

### Requirement 4: Delete Knowledge

**User Story:** As an AI agent, I want to delete outdated or incorrect knowledge artifacts, so that the repository remains accurate and free of stale information.

#### Acceptance Criteria

1. WHEN the Agent invokes the CLI_Tool with an `akr-delete` command and a valid artifact identifier, THE CLI_Tool SHALL remove the Knowledge_Artifact from the Repository and remove the corresponding Embedding from the Vector_Store.
2. WHEN the Agent invokes the CLI_Tool with an `akr-delete` command and an artifact identifier that does not exist, THE CLI_Tool SHALL return an error indicating the artifact was not found.
3. WHEN a Knowledge_Artifact is successfully deleted, THE CLI_Tool SHALL return a confirmation message including the deleted artifact's identifier.

### Requirement 5: Knowledge Artifact Schema

**User Story:** As a developer, I want all knowledge artifacts to follow a consistent schema, so that knowledge is structured, searchable, and machine-readable.

#### Acceptance Criteria

1. THE Knowledge_Schema SHALL require the following fields for each Knowledge_Artifact: title, content, tags, source context, and creation timestamp.
2. THE CLI_Tool SHALL automatically populate the creation timestamp and last-modified timestamp fields when committing or updating a Knowledge_Artifact.
3. THE Knowledge_Schema SHALL support an optional metadata field for arbitrary key-value pairs to allow domain-specific extensions.
4. WHEN a Knowledge_Artifact is committed or updated, THE CLI_Tool SHALL validate the payload against the Knowledge_Schema before persisting.

### Requirement 6: Vector Store Backend

**User Story:** As a developer, I want knowledge artifacts indexed using sqlite-vec for vector search, so that agents can perform fast semantic search over the repository without external database servers.

#### Acceptance Criteria

1. THE Vector_Store SHALL be implemented as an Embedded_Vector_Store using sqlite-vec, running in-process within the Python application and persisting data to local SQLite database files, requiring no separate database server or daemon.
2. THE Vector_Store SHALL generate an Embedding for each Knowledge_Artifact upon commit or update using the configured ONNX_Embedding_Model.
3. WHEN the Agent performs a fetch query, THE Vector_Store SHALL return results ranked by cosine similarity between the query Embedding and stored Embeddings, using sqlite-vec's built-in distance functions.
4. THE Vector_Store SHALL support configurable ONNX_Embedding_Model selection through the Repository configuration, defaulting to a lightweight general-purpose model.
5. IF the configured ONNX_Embedding_Model is unavailable or fails to load, THEN THE CLI_Tool SHALL return an error message identifying the missing model and suggesting corrective action (e.g., pip install command).
6. THE Vector_Store SHALL store all vector data and metadata in SQLite database files using the sqlite-vec extension, which is pip-installable and requires no system-level package installation on the Target_Platform.
7. THE ONNX_Embedding_Model SHALL generate Embeddings locally on CPU without requiring PyTorch, TensorFlow, GPU hardware, or external API keys.

### Requirement 7: Shared Repository Mode

**User Story:** As a team of developers, we want a shared knowledge repository on our system, so that all agents across the team can contribute to and benefit from collective knowledge.

#### Acceptance Criteria

1. THE CLI_Tool SHALL support a Shared_Repository mode where all agents on the system read from and write to a single Repository at a configurable system-level path.
2. WHEN multiple Agents attempt concurrent writes to the Shared_Repository, THE CLI_Tool SHALL serialize writes to prevent data corruption.
3. THE CLI_Tool SHALL use file-system-level locking or an equivalent mechanism to ensure write safety in Shared_Repository mode.
4. WHEN the Shared_Repository path is not configured, THE CLI_Tool SHALL default to a well-known system path (e.g., `/var/lib/agent-knowledge-repo/` on Linux, platform-appropriate equivalents elsewhere).

### Requirement 8: User Repository Mode

**User Story:** As an individual developer, I want a per-user knowledge repository in my `.kiro/` directory, so that I can maintain agent-specific or personal knowledge separate from the shared repository.

#### Acceptance Criteria

1. THE CLI_Tool SHALL support a User_Repository mode where knowledge is stored within the user's `.kiro/knowledge/` directory.
2. WHEN the CLI_Tool is configured for User_Repository mode, THE CLI_Tool SHALL read from and write to only the user-scoped Repository.
3. THE CLI_Tool SHALL allow simultaneous use of both Shared_Repository and User_Repository modes, with a configurable merge strategy for fetch results.
4. WHEN both repository modes are active during a fetch, THE CLI_Tool SHALL return results from both repositories, clearly indicating the source repository for each result.

### Requirement 9: Repository Configuration

**User Story:** As a developer, I want to configure repository settings through a configuration file, so that I can customize behavior without modifying code.

#### Acceptance Criteria

1. THE CLI_Tool SHALL read configuration from a `.kiro/knowledge-config.json` file in the user's home directory or project root.
2. THE CLI_Tool SHALL support configuration of: repository mode (shared, user, or both), shared repository path, embedding model, default fetch result count, and similarity threshold.
3. WHEN a configuration file is not found, THE CLI_Tool SHALL use sensible defaults and operate in User_Repository mode.
4. WHEN the configuration file contains invalid values, THE CLI_Tool SHALL report specific validation errors identifying the invalid fields.

### Requirement 10: Steering File Integration

**User Story:** As a developer, I want steering files that instruct AI agents to use the CLI tool automatically, so that knowledge capture and retrieval happen without manual intervention.

#### Acceptance Criteria

1. THE Steering_File SHALL define rules that trigger the CLI_Tool `akr-fetch` command at the start of each agent interaction to retrieve relevant context.
2. THE Steering_File SHALL define rules that trigger the CLI_Tool `akr-commit` command when the Agent produces a significant learning, code insight, or architectural decision.
3. THE Steering_File SHALL be compatible with existing `.kiro/steering/` directory conventions and not conflict with other Steering_Files.
4. THE Steering_File SHALL include clear instructions for the Agent on when to use akr-commit, akr-fetch, and akr-update commands, using explicit trigger conditions.
5. THE Steering_File SHALL use an `inclusion: auto` directive so that it is automatically loaded into agent context.

### Requirement 11: Steering File Content and Rules

**User Story:** As a developer, I want the steering files to enforce consistent knowledge management patterns, so that all agents on the team follow the same rules for knowledge operations.

#### Acceptance Criteria

1. THE Steering_File SHALL instruct the Agent to tag each Knowledge_Artifact with relevant categories (e.g., "architecture", "bug-fix", "pattern", "dependency").
2. THE Steering_File SHALL instruct the Agent to include source context (file paths, function names, or interaction identifiers) in each committed Knowledge_Artifact.
3. THE Steering_File SHALL instruct the Agent to check for existing similar knowledge before committing a new artifact, to avoid duplication.
4. THE Steering_File SHALL instruct the Agent to update existing Knowledge_Artifacts when new information supersedes prior knowledge rather than creating duplicates.

### Requirement 12: List and Browse Knowledge

**User Story:** As an AI agent or developer, I want to list and browse knowledge artifacts by tags or date, so that I can explore the repository beyond semantic search.

#### Acceptance Criteria

1. WHEN the Agent invokes the CLI_Tool with an `akr-list` command, THE CLI_Tool SHALL return a paginated list of Knowledge_Artifacts sorted by last-modified timestamp in descending order.
2. WHEN the Agent invokes the CLI_Tool with an `akr-list` command and a `--tags` filter, THE CLI_Tool SHALL return only Knowledge_Artifacts matching all specified tags.
3. WHEN the Agent invokes the CLI_Tool with an `akr-list` command and a `--since` date filter, THE CLI_Tool SHALL return only Knowledge_Artifacts modified on or after the specified date.
4. THE CLI_Tool SHALL support a `--limit` option on the `akr-list` command to control the number of results returned, defaulting to 20.

### Requirement 13: Knowledge Artifact Serialization

**User Story:** As a developer, I want knowledge artifacts to be serialized to and from a structured format, so that they can be stored, transmitted, and inspected reliably.

#### Acceptance Criteria

1. THE CLI_Tool SHALL serialize Knowledge_Artifacts to JSON format for storage and transmission.
2. THE CLI_Tool SHALL deserialize JSON payloads into Knowledge_Artifact objects for processing.
3. THE Pretty_Printer SHALL format Knowledge_Artifact objects into human-readable JSON with consistent indentation.
4. FOR ALL valid Knowledge_Artifact objects, serializing then deserializing SHALL produce an equivalent object (round-trip property).

### Requirement 14: Minimal External Dependencies and RHEL 9 Native Deployment

**User Story:** As a developer on a RHEL 9 system, I want the CLI tool to be written in Python with all dependencies pip-installable, so that I can deploy and use it without installing additional system-level packages or external database servers.

#### Acceptance Criteria

1. THE CLI_Tool SHALL be implemented in Python (compatible with Python 3.9+, available natively on RHEL 9 or as an app stream).
2. THE CLI_Tool SHALL prefer Standard_Library capabilities over external libraries for all functionality where a Standard_Library solution exists.
3. THE CLI_Tool SHALL be distributable as a pip-installable Python package that runs on the Target_Platform without requiring installation of additional system-level packages.
4. THE CLI_Tool SHALL NOT require any external database server, message broker, or daemon process to operate.
5. THE Embedded_Vector_Store SHALL use sqlite-vec (pip-installable) with the SQLite engine available natively on RHEL 9, requiring no separately installed database server.
6. THE CLI_Tool SHALL use a lightweight ONNX-based embedding library (pip-installable) as the ONNX_Embedding_Model for local Embedding generation, requiring no PyTorch, TensorFlow, GPU hardware, or external API keys.
7. IF an external library is required for functionality not available in the Standard_Library, THEN THE CLI_Tool SHALL ensure the library is installable via pip without requiring system-level package installation (e.g., `dnf install` or `yum install`) on the Target_Platform.
8. THE CLI_Tool SHALL run on RHEL 9 using only the Python runtime and pip-installable packages, with no system-level dependencies beyond Python itself.
9. WHEN the CLI_Tool is installed on a clean RHEL 9 system with only the Python runtime present, THE CLI_Tool SHALL be fully operational after running `pip install` alone.
